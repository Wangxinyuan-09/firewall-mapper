## 重新设计：访问关系 = 流向卡片 + 智能联动筛选

### 一、流向卡片（核心展示）

不再按"策略/NAT 一条条列"，而是按 **流向（Flow）** 聚合。一条流向 = `源 → 目的`，把相关的 NAT 端口、策略服务、命中规则汇总到一张卡片里。

示例：外网 → WAF

```text
┌─────────────────────────────────────────────────────────────┐
│  any (untrust)          ──DNAT──▶          waf (dmz)        │
│  外网客户                                    10.0.0.5        │
│                                                              │
│  入口端口聚合（3 条 DNAT）：                                  │
│    203.0.113.7:tcp/443  →  waf:tcp/8443     #3              │
│    203.0.113.7:tcp/80   →  waf:tcp/8080     #4              │
│    203.0.113.8:tcp/22   →  waf:tcp/22       #5  [log]       │
│                                                              │
│  放行策略（按服务聚合）：                                     │
│    ✅ tcp/8443, tcp/8080   ← 策略 #12 permit                 │
│    ✅ tcp/22               ← 策略 #13 permit                 │
│                                                              │
│  结论：3 个端口全部有 permit 策略覆盖 ✓                      │
└─────────────────────────────────────────────────────────────┘
```

WAF → web-pool（无 NAT 直连）：

```text
┌─────────────────────────────────────────────────────────────┐
│  waf (dmz)             ──直连──▶           web-pool (trust) │
│                                                              │
│  放行策略（按服务聚合）：                                     │
│    ✅ tcp/8080, tcp/8443   ← 策略 #20 permit                 │
│    ❌ tcp/3306             ← 策略 #25 deny (优先)            │
│    ✅ tcp/3306             ← 策略 #30 permit (被 #25 遮蔽)   │
└─────────────────────────────────────────────────────────────┘
```

孤儿 DNAT：

```text
┌─────────────────────────────────────────────────────────────┐
│  any                   ──DNAT──▶           db-pool          │
│    203.0.113.9:tcp/3306 → db-pool:tcp/3306   #7             │
│  ⚠ 无任何 permit 策略覆盖 tcp/3306 → 端口开了但不通          │
└─────────────────────────────────────────────────────────────┘
```

### 二、智能联动筛选（重点优化）

放弃"3 个独立下拉 + 全量对象列表"的做法。改为 **基于真实关系收敛候选**：

**字段：**
- 源（可空）
- 目的（可空）
- 服务（可空）

**联动规则**——下拉里只显示在配置里实际出现过、且与已选项构成真实流向的选项：

1. 初始状态：
   - 源下拉 = 所有真实作为"流向源"出现过的对象（来源：策略 `srcAddr` ∪ 含义为外部入口的 `any`）
   - 目的下拉 = 所有真实作为"流向目的"出现过的对象（来源：策略 `dstAddr` ∪ NAT `translatedPool`）
   - 服务下拉 = 所有真实出现过的 service / port 字面（来源：策略 service ∪ NAT 端口）
2. 选了"源 = A" 之后：
   - 目的下拉收敛为：与 A 存在至少一条流向（策略或 NAT 链）的对象集合
   - 服务下拉收敛为：这些流向里实际出现过的服务/端口
3. 选了"目的 = B" 之后：源、服务同理收敛
4. 选了"服务 = tcp/443" 之后：源、目的收敛到出现过该服务的流向
5. 任一字段右侧显示"剩余 N 项"，可一键清除
6. 每个下拉项后面带个小灰字角标，标出"出现在 X 条流向中"，让用户预判选完后会剩多少
7. 支持搜索框过滤；支持字面输入 IP / `tcp/443`（沿用 `resolveEndpoint`），输入后同样按联动规则收敛剩余两个字段

**对照表**

| 旧做法 | 新做法 |
|---|---|
| 3 个 select 各自独立 | 三向联动，互相收敛 |
| 下拉列出全部 addresses / address-groups | 只列出真实出现在流向里的对象 |
| 选完才知道有没有匹配 | 选项本身已被预过滤，永远不会空 |
| 任意组合允许，多数返回 0 结果 | 不可能组合在 UI 上根本选不到 |

### 三、顶部筛选（与卡片列表的关系）

- 三个联动 select 留空 = 显示全部流向卡片
- 任一字段有值 = 只显示匹配该字段的流向卡片
- 额外两个 chip：`只看 DNAT 流向` `只看孤儿/缺口（异常）`
- 排序：异常优先 → DNAT 流向 → 字典序

去掉 Tabs、ChainView、独立的"命中策略表 / 命中 NAT 表"——所有信息都进卡片。

### 文件改动

- `src/lib/access.ts` 新增：
  - `buildFlows(cfg): Flow[]`——枚举 `(src, dst)` 对，组装 `dnatEntries`、`policySegments`、`coverage`。
  - `buildFlowIndex(flows)`——给联动筛选用，返回 `{sources, destinations, services, counts}` 以及 `narrowBy({src?, dst?, svc?})` 收敛函数。
  - 类型：`Flow` / `FlowDnatEntry` / `FlowPolicySegment` / `Coverage`。
- `src/routes/access-graph.tsx`：完全重写
  - 顶部 3 个联动 select（基于 `FlowIndex.narrowBy`）+ 2 个 chip
  - `<FlowCard>` 列表
  - 复用 `Badge` / `LineLink` / `ObjectName` / `summarizeService` / `classifyIntermediary`
- 不改解析器、不改其他页面、不引入图形库。

### 不做

- 不做模拟下发 / 真实抓包
- 不做 SNAT 卡片（SNAT 留在 NAT 页）
- 不做图论可视化
